Teoria

Iniciando o que foi esplicado pelo curso, passaremos pela breve explicação teórica do que são bancos de dados não relacionais e um pouco das especifícações do nosso querido MongoDB.

NoSQL

O termo NoSQL é um termo relativamente novo para uma idéia já antiga dos bancos de dados. Feraamentas de armazenamento não baseadas na estrutura relacional. Contudo tecnologias que fazem parte da classe NoSQL estão longe de serem substitutas do modelo relacional, já tão consolidado no mundo.

NoSQL referenciam-se tecnologias de armazenamento que não se baseam obrigatóriamente em esquemas de dados predefinidos e possuem capacidade de armazenar estruturas de dados específicas, complementando assim o modelo generalista e estruturado de bancos de dados relacionais

Existem alguns bancos de dados não relacionais bastante disseminados atualmente pelo seu potêncial de aplicação em problemas específicos. Exitem alguma categorias de bancos, então é necessário realizar uma análise prévia para se descobrir qual categoria de solução melhor atende as necessidades de desenvolvimento. Das categorias principais temos:

  • Orientados a chave-valor
  • Orientados a documentos
  • Orientados a grafos
  • Orientados a colunas

Chave-valor

Baseado na ideia de armazenar cada valor em uma chave, e só necessitar buscar, ordenar e gerenciar estas chaves tornando a busca e recuperação dos dados bastante eficiente, além de possuir uma estrutura bastante simples.

Mas esta simplicidade tem seu preço. Este tipo de estrutura não permite se trabalhar com os valores internos de um registro mesmo que este seja um objeto complexo e só é possível acessar determinado valor se soubermos previamente a chave daquele registro.

Documentos

Semelhante ao modelo de chave valor, mas por outro lado supre a necessidade de análise dos valores intenos dos objetos, considerados como os documentos dessas bases de dados. É necessário se especificar uma estrutura de documentos a serem armazenados, o MongoDB, objeto principal desta página é orientado a documentos json.

Como desvantagem este modelo apresenta pouca usabilidade em modelos que necessitam de grande cobertura de ligação e dependência dos objetos armazenados por ter o foco no armazenamento completo da informação e para se armazenar vínculos é necessário muitas vezes se armazenas ambos os documentos repetidas vezes.

Grafo

Baco de dados focados em armazenamento de dados que possuam grande dependência entre sí por se basear na teoria de grafos onde os dados seriam o nódulos e o relacionamento destes seriam as suas arestas. Altamente indicados para aplicações onde é necessário o armazenamento dos dados, como em uma rede social.

Por outro lado a estrutura de grafos se mostra demasiada complexa para modelos onde a informação se encontra em objetos isolados e o relacionamento entre estes é pouco importante. Nestes casos podem ser utilizados os bancos relacionais ou os oientados a documentos

Colunas

Modelo baseado em colunas assemelha-se ao modelo relacional só que de uma maneira tridimensional por adicionar um tipo a mais de filtro que facilita no gerenciamento de grandes volumes de dados.

Tornou-se famoso com o projeto Big Table da Google focado em big data para o gerenciamento e escalabilidade de grandes fontes de dados

MongoDB

MongoDB é um sinônimo para gigantesco e o nome vem com o objetivo de trazer a mensagem que a ferramenta passa com o seu poder de escabilidade e alta performance.A aplicação foi desenvolvida em C++ e possui estrutura baseada em documentos json.

Escabilidade

O MongoDB tem como característica marcante a otimização atravez da escabilidade horizontal, que nada mais é o armazenamento distribuido dos dados em diversos servidores, buscando a dispersão dos dados acima do poder de processamento dos hosts de bancos de dados

Scheemaless

Este SGBD possui a característica comum em bancos de dados não relacionais que é a dispensabilidade de estrutura no armazenamento dos dados previamente ao armazenamento. Ao contrario do quese imagina é possível manter a organização dos dados mesmo em ambientes desta maneira por definir a estrutura dos dados direto nas aplicações.

Memory-mapped files

Esta abordagem deste SGBD utiliza a memória volátil para o armazenamento e gerenciamento dos dados. Isto possibilita uma maior fluência das informações pois os dados persistidos em memória fisica são mais lentos do que os utilizados em memória virtual e neste tipo de abordagem são utilizados apenas quando é realmente necessário. Este paradigma só é eficientemente aplicado graças as cappedcollections

Capped Collections

Capped collections são coleções de tamanho fixo que são alocadas no disco do host da aplicação antes mesmo de qualquer utilização daquele espaço de armazenamento. Por alocar este espaço préviamente o MongoDB pode inserir os dados de forma sequencial com isso aumentando a performance de manipulação dos dados. Ao se extrapolar o espaço de armazenamento de uma capped collection ela passa a sobreescrever o dados mais antigos.
